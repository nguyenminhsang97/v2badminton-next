# S1-B1: Schemas cho core business data

## Objective

Define 5 Sanity schemas cho data business-critical đang có sẵn trong codebase. Sau ticket này, Sanity Studio hiển thị đầy đủ 5 document types với đúng fields, types, và validations theo content models v2 spec.

## Input

- Content models v2 spec (đã chốt trong conversation — xem bảng dưới)
- Data sources hiện tại để tham khảo field types:
  - `src/lib/site.ts` → site_settings
  - `src/lib/locations.ts` → location
  - `src/lib/pricing.ts` → pricing_tier
  - `src/lib/schedule.ts` → schedule_block
  - `src/lib/faqs.ts` → faq

## Acceptance Criteria

1. Mở Sanity Studio → thấy 5 document types trong sidebar
2. Click vào mỗi type → form hiển thị đúng fields
3. `site_settings` chỉ cho tạo 1 document (singleton pattern)
4. `location.district` hiển thị dropdown với 2 options
5. `pricing_tier.kind` hiển thị dropdown với 3 options
6. `schedule_block.location` hiển thị reference picker, chỉ cho chọn location documents
7. `schedule_block.timeSlotId` là free-text string (không phải dropdown) — valid values trong edge case note
8. `faq.pages` cho chọn multiple values
9. Slug fields tự generate từ name/question
10. Boolean fields default true cho isActive
11. `pricing_tier`: conditional fields ẩn đúng theo kind — group fields ẩn khi kind = private/enterprise

## Edge Cases

- `site_settings` singleton: Sanity không có singleton type native — cần dùng pattern `document type + initial value + hide from create` hoặc plugin
- `pricing_tier` có 3 kinds với fields khác nhau: `pricePerMonth`/`sessionsPerMonth`/`groupSize` chỉ relevant cho group, `pricePerHour` chỉ cho private — dùng conditional hidden fields trong Studio để tránh operator nhầm lẫn
- `displayPrice` là required string cho tất cả tiers — không phải optional override. Group tiers render "X VNĐ / tháng", private "X VNĐ / giờ", enterprise "Giá thương lượng"
- `faq.answer` nên là block content (Sanity Portable Text) thay vì plain text — để sau này có thể bold, link, format
- `schedule_block.levelLabels` **không lưu vào Sanity** — field này redundant, frontend sẽ derive từ `levels` theo map: `co_ban → "Cơ bản"`, `nang_cao → "Nâng cao"`, `doanh_nghiep → "Doanh nghiệp"`. Operator chỉ cần chọn `levels`, không cần nhập labels thủ công.
- `schedule_block.timeSlotId` phải khớp chính xác với `TimeSlotId` type trong `schedule.ts` — đây là contract với frontend prefill logic

## Non-Scope

- Chưa nhập data (S1-C group)
- Chưa define coach, testimonial (S1-B2)
- Chưa define money_page, campaign, post (S1-B3)
- Chưa custom Studio UI (ordering, previews, etc.)

## Risk / Lưu ý

- Schema changes sau khi đã có data sẽ phức tạp hơn — nên review kỹ fields trước khi migrate data
- Portable Text (block content) cho FAQ answer có thể phức tạp hơn plain text khi render ở frontend — nhưng đáng trade-off vì linh hoạt hơn
- Đặt tên schema files nhất quán: 1 file per schema, đặt trong folder rõ ràng
